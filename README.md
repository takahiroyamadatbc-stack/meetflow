# MeetFlow

麻雀コミュニティ向けの、**空き予定と開催条件のマッチングでイベントを成立させる**予定調整プラットフォーム。

「調整さん」のように全員の回答を可視化する方式でも、Calendlyのように自分の予定を一方的に公開する方式でもなく、**個人の空き予定は誰にも公開せず、システムが導き出した候補だけを関係者全員に同時提示する**「相互非公開型マッチング」という設計を軸にしています。

> 個人開発中のプロダクトです。設計・実装の考え方を見てもらう目的で公開しています。

## 背景・課題意識

複数人でのメンバー参加型イベント（麻雀の対局など）の日程調整は、既存のツールだと以下のような駆け引き・後出しが起きがちです。

- 「調整さん」型：全員の回答が可視化されるため、周りの回答を見てから自分の回答を決める動きが起きる
- Calendly型：自分の予定を一方的に公開する前提で、対等な複数人の調整には向かない

MeetFlowでは、ユーザーは空き時間だけを登録し、コミュニティ管理者はイベントテンプレート（ゲーム種別・人数範囲・優先度）を登録します。システムはそれらを突き合わせてスコア付きの候補を生成しますが、**候補を自動確定することは決してなく**、必ず管理者のレビュー・承認を経ます（human-in-the-loop）。「システムが決定する」挙動は、この設計における最大の禁じ手として扱っています。

初期ドメインは麻雀コミュニティ（四人麻雀・三人麻雀）ですが、データモデルは意図的にゲーム非依存に保ってあり、ポーカーやボードゲームなど他ジャンルへの展開も見据えています。

## 主な機能

- コミュニティ作成・招待URL発行・参加承認
- 空き予定のカレンダー登録（バッチ登録対応）
- イベントテンプレート（開催条件）の登録
- マッチング候補の自動生成 → 管理者承認によるイベント確定
- イベントライフサイクル管理（参加確定、キャンセル申請と承認、イベント全体中止）
- ダブルブッキング防止（同期ハードブロック＋非同期の衝突警告の二層構成）
- 対局結果の記録・集計
- アプリ内通知＋Webプッシュ通知（PWA対応）
- 操作ログ（誰が何をしたかの監査証跡）

## アーキテクチャ

```
User → CloudFront → S3 (React SPA)
                   ↘
                     API Gateway (Cognito Authorizer) → 7ドメインLambda → DynamoDB（単一テーブル）
```

- 認証はAmazon Cognito。JWT検証はAPI GatewayのCognito Authorizerで完結し、独自の認証Lambdaは持たない
- Lambdaはエンドポイント単位ではなく **ドメイン単位（User/Community/Availability/Matching/Event/Result/Notification の7つ）** で分割
- DynamoDBは単一テーブル設計。GSIは2本のみで、オーバーロードGSIパターンを採用（事前に洗い出したアクセスパターン集合の上に構築しており、場当たり的なクエリ追加はしない）
- EventBridgeは通知・コミュニティ横断の衝突検知など後続処理のファンアウトにのみ使用し、マッチング処理自体の自動起動には使わない

| CI | GitHub Actions（backend-test / frontend-ci / infra-synth を並列実行） |

## 設計上のこだわり

- **Human-in-the-loopの徹底**：マッチング候補の生成と確定を明確に分離し、確定は必ず管理者操作を経由する
- **相互非公開の維持**：他メンバーの生の空き予定を返すAPI・UIを持たない。関係者に見えるのはマッチング後の候補のみ
- **ダブルブッキング防止の非対称設計**：イベント確定時は同期的なハードチェックでブロックする一方、コミュニティ横断の事後衝突検知は管理者向けの警告表示に留め、自動除外はしない
- **公平性指標・衝突警告はあくまで表示専用シグナル**：スコアリングや自動除外ロジックには組み込まない
- **単一テーブル設計への忠実さ**：PK/SK/GSIの設計から逸脱するアドホックなクエリを禁止し、DynamoDB物理設計書のアクセスパターン一覧を都度参照
- **IAM最小権限**：各Lambdaの権限をLambda設計書の方式に従って個別に絞り込み
- **Docker非依存のネイティブ依存バンドリング**：Web Push送信に必要なネイティブ拡張（cryptography等）を、`ILocalBundling`によるローカルバンドリングでDocker無しでLambda Layer化

## 開発プロセス

要件定義書・機能要件書・API設計書・DynamoDB物理設計書・Lambda設計書などをバージョン管理しながら実装前に必ず参照する運用にしています。各ドキュメントは機能ID（F-xxx）で相互参照されており、ある機能の実装時は要件〜設計〜実装をトレースできるようにしています。

- バックエンドはpytest + moto（DynamoDBモック）によるユニットテストを7ドメイン全てに整備
- CIはGitHub Actionsでbackend-test / frontend-ci / infra-synthを並列実行
- 変更はGitHub Issueで管理し、Conventional Commits形式でコミット

## 開発状況

バックエンドは7ドメインLambda・DynamoDB単一テーブル・API Gateway・Cognito・CloudFront配信までを実装し、実際にHTTP経由で呼び出し可能な状態です。フロントエンドも認証・コミュニティ・空き予定に加え、マッチング・イベント・成績・通知・PWA対応まで一通りの画面を実装済みです。CD（GitHub Actionsからの自動デプロイ）は未整備で、現状は手動デプロイで運用しています。
