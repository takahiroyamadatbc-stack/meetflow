## 1. 目的

本書はMeetFlowにおける機能要件を定義する。対象機能、利用者、処理概要、入力情報、出力情報を明確化し、後続工程である画面設計・API設計・AWS実装設計の基礎とする。

---

## 2. システム利用者

### 2.1 ユーザー

一般利用者。利用可能機能：アカウント管理、プロフィール管理、コミュニティ参加、空き予定登録、イベント参加、成績確認

### 2.2 コミュニティ管理者

コミュニティ運営者。利用可能機能：コミュニティ管理、メンバー管理、開催条件設定、マッチング候補確認、イベント承認

## 3. 認証・ユーザー管理機能

### F-001 ユーザー登録

**概要**：ユーザーがMeetFlowを利用するためのアカウントを作成する。

**利用者**：未登録ユーザー

**入力**：メールアドレス、パスワード、ニックネーム

**処理**：
1. Cognito User Poolへの認証情報登録
2. ユーザーID発行
3. ユーザープロフィール作成（DynamoDB）

**出力**：登録完了

### F-002 ログイン

**概要**：登録済みユーザーがログインする。

**入力**：メールアドレス、パスワード

**処理**：認証成功後、アクセストークン発行。

**出力**：ログイン状態

### F-003 プロフィール管理 **[v1.7修正、v1.8修正]**

**概要**：ユーザー情報を管理する。

**管理項目**：ニックネーム、アイコン、自己紹介、対応可能ゲーム、初心者対応可否、参加頻度上限（ゲームジャンル単位、回数＋期間：週/月、任意）[v1.7追加]、自動承認設定（イベント仮確定後の参加承認を以降自動で行うかどうか、任意・既定false）[v1.8追加]

**将来追加**：趣味、得意ジャンル、信頼スコア

> **[v1.7追加]** 参加頻度上限は、体力・満足度等の理由による個人ごとのペース調整を目的とした設定で、マッチングのスコアリング（F-403）に反映される。詳細はF-109（コミュニティ単位の上書き）およびF-403を参照。
>
> **[v1.8追加]** 自動承認は、F-502bの参加承認をユーザー本人に代わって都度の操作なしに行うための事前オプトイン設定。詳細はF-109b（コミュニティ単位の上書き）およびF-502bを参照。システムが自律的にイベントを確定するわけではなく、あくまで本人が事前に与えた同意の反映である（要件定義書v1.7 §17/§19）。

## 4. コミュニティ機能

### F-101 コミュニティ作成

**概要**：ユーザーが新しいコミュニティを作成する。

**利用者**：ユーザー

**入力**：コミュニティ名、説明、ジャンル、**メンバー承認要否フラグ（memberApprovalRequired）** [v1.1追加]

**処理**：
1. コミュニティ作成
2. 作成者をOWNERとして登録

### F-102 招待URL発行

**概要**：コミュニティ参加用URLを発行する。

**利用者**：OWNER / ADMIN

**処理**：招待トークン生成。

**出力**：招待URL（例：`https://meetflow.com/invite/xxxxx`）

### F-103 コミュニティ参加 **[v1.1修正]**

**概要**：招待URLからコミュニティへ参加する。

**入力**：招待URL

**処理**：コミュニティの `memberApprovalRequired` 設定に応じて分岐する。

- **即時参加モード**（`memberApprovalRequired = false`）：招待情報確認 → ユーザー登録 → MEMBERとして即時追加
- **承認制モード**（`memberApprovalRequired = true`）：招待情報確認 → 参加リクエスト（JoinRequest）を作成しPENDING状態にする → 管理者へ通知 → 管理者が承認/却下（F-104b, F-104c参照）

### F-104 メンバー管理 **[v1.1修正]**

**概要**：コミュニティメンバーを管理する。

**利用者**：OWNER / ADMIN

**機能**：
- メンバー一覧表示
- 権限変更
- 退出処理（強制退会）
- **一時停止**（v1.0機能要件書で漏れていたため追加）
- **募集参加制限**（同上）

**[v1.9追加]** 退出処理（強制退会）は、対象メンバーがそのコミュニティ発のイベントで開催日時が未来のCONFIRMED（または仮確定時点で予約済み扱いとなるAWAITING_APPROVAL）な参加を1件でも持つ場合はブロックする（`409 MEMBER_HAS_UPCOMING_EVENTS`）。強制退会後もParticipantレコードだけが残り、非メンバーが確定参加者であり続ける不整合を防ぐため（Issue #26。F-104dと同じチェックロジックを流用）。

### F-104b 参加リクエスト承認 **[v1.1新規]**

**概要**：承認制コミュニティへの参加リクエストを管理者が承認する。

**利用者**：OWNER / ADMIN

**処理**：JoinRequestを承認済みに変更 → Membership作成（MEMBER登録）→ OperationLog登録

### F-104c 参加リクエスト却下 **[v1.1新規]**

**概要**：参加リクエストを却下する。

**利用者**：OWNER / ADMIN

### F-104d メンバー自主退会 **[v1.9新規]**

**概要**：メンバー自身の意思でコミュニティを退会する。

**利用者**：MEMBER / ADMIN / OWNER（OWNERは唯一のOWNERである間は退会できない）

**処理**：Membership削除 → OperationLog登録（`LEAVE_COMMUNITY`。管理者による強制退会`REMOVE_MEMBER`とは区別する）

**制約**：
- 唯一のOWNERは退会できない（`409 LAST_OWNER_CANNOT_LEAVE`。先にF-106でオーナー移譲が必要）
- そのコミュニティ発のイベントで開催日時が未来のCONFIRMED（またはAWAITING_APPROVAL）な参加を1件でも持つ場合は退会できない（`409 MEMBER_HAS_UPCOMING_EVENTS`）。ブロックされた場合は先にイベント参加のキャンセル申請（F-601）を案内する。自動キャンセル等の副作用は起こさない（要件定義書v1.7 §17/§19のヒューマン・イン・ザ・ループ原則）

### F-105 権限管理

| 権限 | 内容 |
|---|---|
| OWNER | 全操作可能 |
| ADMIN | 運営操作可能 |
| MEMBER | 一般利用 |

### F-106 管理者変更 **[v1.1修正]**

**概要**：コミュニティ管理者を変更する。

**利用者**：OWNER

**処理**：OWNER権限を別ユーザーへ移譲する。**移譲後、元OWNERはMEMBER権限に戻る**（ADMINへの自動降格ではない）。

## 5. 空き予定管理機能

### F-201 空き予定登録

**概要**：ユーザーが参加可能な日時を登録する。

**入力**：開始日時、終了日時、対応ゲーム、コメント

例：`2026/07/20 19:00〜23:00 四麻・三麻可能 初心者OK`

> **[v1.1補足]** 本機能は単発予定のみを対象とする。繰り返し予定（要件定義書11.2）はPhase2で別途機能定義する。

### F-202 空き予定編集

**概要**：登録済み予定を変更する。

**更新可能項目**：日時、ゲーム条件、コメント

### F-203 空き予定削除

**概要**：不要になった予定を削除する。

### F-204 空き予定一覧表示

**概要**：**自分専用**の一覧表示であり、自身の登録済み予定のみを表示する。他メンバーの空き予定を閲覧する機能・APIは存在しない。

**表示項目**：日時、状態、対応ゲーム、関連イベント

> **[v1.4追加]** 本機能は要件定義書v1.4 29章「相互非公開型マッチング」の原則に基づくものである。空き予定という個人の入力情報は本人以外に公開せず、システムが導き出した候補（F-401〜F-404）のみが関係者に共有される。この非対称性のない構造により、正直な予定登録に不利益が生じない設計とする。

### F-205 空き予定提出リクエスト作成 **[v1.3新規]**

**概要**：管理者が対象メンバーに対して、空き予定の提出を依頼する。要件定義書v1.3 28章に対応。

**利用者**：OWNER / ADMIN

**入力**：対象期間（開始/終了）、提出期限、対象範囲（全員 / 指名リスト）、メッセージ（任意）

**処理**：

1. AvailabilityRequest作成
2. 対象メンバーへ通知作成（F-701連携）

### F-206 未提出メンバー確認 **[v1.3新規]**

**概要**：管理者が、あるリクエストについて対象期間内の空き予定をまだ登録していないメンバーの一覧を確認する。

**利用者**：OWNER / ADMIN

**処理**：対象メンバー一覧取得 → 各メンバーについてAvailabilityの既存GSI1（`USER#{userId}`, `AVAIL#{startTime}`が対象期間内か）を確認 → 該当レコードが無いメンバーを一覧表示

> **[v1.3補足]** あくまで「対象期間内に空き予定レコードが存在するか」の軽量な有無判定であり、督促対象を洗い出す補助機能に留める。自動催促・提出強制は行わない。

## 6. 共通制御

**権限チェック**：すべての機能で以下を確認する。
- ログイン済みか
- 対象コミュニティ所属済みか
- 操作権限があるか

**エラー処理**（共通エラー）

| エラー | 内容 |
|---|---|
| USER_NOT_FOUND | ユーザー不存在 |
| COMMUNITY_NOT_FOUND | コミュニティ不存在 |
| FORBIDDEN | 権限不足 |
| INVALID_PARAMETER | 入力不正 |

> **[v1.1補足]** 上記は共通エラーとし、招待URL無効・定員オーバー・二重登録などのドメイン固有エラーは各機能セクションで個別に定義する（次版で追記予定）。

## 7. 開催条件管理機能

### F-301 開催条件作成

**概要**：コミュニティ管理者がイベント開催条件を登録する。ユーザー募集ではなく、マッチング条件として利用する。

**利用者**：OWNER / ADMIN

**入力項目**：イベントジャンル、ゲーム種別、開催可能期間、最低人数、最大人数、必須メンバー、希望条件、優先度

例：ゲーム＝四人麻雀 / 人数＝5〜6人 / 条件＝初心者歓迎 / 優先度＝80%

### F-302 複数開催条件登録

**概要**：1つの開催目的に対して複数パターンを設定できる。

例：条件1（四麻・5〜6人・初心者歓迎・優先度70%）、条件2（四麻・5〜6人・優先度20%）、条件3（三麻・3人・優先度10%）

**システム動作**：複数条件を比較し、成立可能性と優先度から候補生成する。

### F-303 開催条件編集

**変更項目**：人数、ゲーム種別、条件、優先度

### F-304 開催条件削除

不要な条件を削除する。

## 8. マッチング候補生成機能

MeetFlowの主要機能。

### F-401 マッチング実行

**概要**：空き予定と開催条件を照合し、イベント候補を生成する。

**実行契機**：
- 管理者による手動実行（MVP）
- 空き予定追加時（将来）
- 定期実行（将来）

> **[v1.1補足]** MVPでは手動実行のみを提供するが、内部処理は「開催条件（テンプレート）単位の実行」として抽象化し、将来の定期実行・自動トリガーからも同一ロジックを再利用できる構成とする（実装方針としてAPI設計書6.1参照）。

**処理フロー**：開催条件取得 → 対象コミュニティ取得 → ユーザー空き予定取得 → 条件照合 → 候補生成 → スコア計算 → 候補保存

### F-402 条件判定 **[v1.1修正、v1.6修正]**

**必須条件**：
- 同一コミュニティ所属
- 日時一致（**全員の空き予定に共通する時間帯が存在すること**。開始・終了時刻の完全一致ではない [v1.6修正]）
- 人数条件（**最低〜最大人数の範囲内であること**。厳密な一致ではない）
- 必須メンバー条件一致

> **[v1.6追加]** 「日時一致」の実装が当初、空き予定の開始・終了時刻の完全一致でのみ候補を生成しており、共通する時間帯があっても不成立になる不具合があった（例：Aが11:00〜13:00、Bが12:00〜14:00の場合、共通区間12:00〜13:00で成立すべきところ不成立だった）。全員の空き予定区間の交差（重複区間）判定であることを明記する。要件定義書v1.5 13.1も同時に更新。

### F-403 スコアリング **[v1.1修正]**

候補ごとに評価点を算出する。

**加点要素**

| 項目 | 内容 |
|---|---|
| 条件一致 | 希望条件一致 |
| 未参加期間 | 最近参加していない |
| 初心者対応 | 対応可能者 |

> **[v1.1修正]** 「信頼度」はMVP対象外（Phase2）のため加点要素から除外する。将来Phase2で信頼スコア・NG設定を組み込めるよう、スコア計算処理は関数として分離しておくことを推奨する。

**減点要素**

| 項目 | 内容 |
|---|---|
| 参加頻度上限超過 [v1.7追加] | 対象メンバーが、設定した参加頻度上限（F-003/F-109、集計期間内・同一ジャンルでコミュニティ横断カウント）にすでに達している場合に減点する |

> **[v1.7追加]** 参加頻度上限超過は、候補からの機械的な除外には使用しない。減点した上でF-404の成立理由に「〇〇さんは〈期間〉の上限（△回）にすでに達している」旨を明記し、最終判断は管理者に委ねる（要件定義書v1.6 30章、19章のヒューマン・イン・ザ・ループ原則）。F-404の**[v1.2追加]**にある「衝突警告」「公平性指標」（表示専用・スコアには使用しない）とは異なり、こちらは実際にスコア計算へ反映される点に注意。

**減点要素（将来）**：キャンセル率、相性、偏り

### F-404 候補一覧表示 **[v1.2修正]**

表示内容例：候補A / 日時7/20 19:00 / ゲーム四麻5人 / メンバーA,B,C,D,E / 評価92点 / 理由（全員予定一致・初心者対応可能・優先条件一致）

> **[v1.2追加]** 候補・メンバーごとに以下の参考情報を付加表示する（スコアリングや自動除外には使用しない）。
> - **衝突警告**：他コミュニティで確定した別イベントと時間重複が検知された場合の警告表示
> - **公平性指標**：メンバーごとに「直近30〜60日で候補に入ったが確定しなかった回数」を表示し、特定メンバーへの偏りへの気づきを支援する

## 9. イベント管理機能

### F-501 イベント作成 **[v1.1修正]**

**概要**：マッチング候補からイベントを作成する。

**作成方法**：候補選択 / 手動作成

**入力**：候補ID、**会場（locationId、または自由記述のlocationNote）** [v1.1追加]

> **[v1.1追加]** 前回決定「イベント作成時に管理者が会場を選択する」を反映。会場はコミュニティに紐づく会場マスタ（F-107参照）から選択するか、自由記述で補足する。

### F-107 会場管理 **[v1.1新規]**

**概要**：コミュニティに紐づく「よく使う会場」をマスタとして登録・編集・一覧表示する。

**利用者**：OWNER / ADMIN

**入力項目**：会場名、住所（任意）、補足コメント（例：全自動卓あり）

**機能**：会場登録、会場編集、会場一覧取得

### F-108 コミュニティごとの表示名 **[v1.5新規]**

**概要**：ユーザーはコミュニティごとに、そのコミュニティ内で表示される名前（displayName）を個別に設定できる。未設定の場合は、プロフィール（F-003）のニックネームがそのまま表示される（フォールバック）。この方式により、既存ユーザーは何も設定しなくても従来通り動作する。

**利用者**：コミュニティの全メンバー（自分自身の表示名のみ設定可能）

**背景**：ユーザーが複数コミュニティに所属する場合、コミュニティによって呼ばれ方が異なる実態（例：麻雀仲間では「たか」、別サークルでは本名寄り）に対応するため。

**適用範囲**：メンバー一覧（F-104）、マッチング候補のメンバーカード表示、イベント参加者一覧、成績登録画面など、コミュニティ文脈で他メンバーの名前を表示するすべての箇所。プロフィール画面（自分自身の表示）およびコミュニティに属さない文脈では、引き続きグローバルなUser.nicknameを表示する。参加リクエスト一覧（承認画面）の申請者は、承認されるまでMembership自体が存在しないため対象外（常にUser.nicknameを表示）。

**重複チェック**：なりすまし・混同防止のため、displayName設定・変更時に同一コミュニティ内での重複をチェックする。チェック対象は「他メンバーのdisplayName」および「他メンバーの実効表示名（displayName未設定者のUser.nickname）」の両方とする。重複していた場合は`409 DISPLAY_NAME_ALREADY_TAKEN`を返す。

**入力項目**：表示名（30文字以内、任意。空欄で解除しUser.nicknameへのフォールバックに戻す）

**対応API**：`PUT /communities/{communityId}/members/me/display-name`

### F-109 参加頻度上限（コミュニティ単位の上書き） **[v1.7新規]**

**概要**：ユーザーはコミュニティごとに、F-003で設定した参加頻度上限（回数＋期間）を個別に上書きできる。未設定の場合は、プロフィール（F-003）側の全体設定にフォールバックする。表示名（F-108）と同じフォールバック構造。

**利用者**：コミュニティの全メンバー（自分自身の設定のみ変更可能）

**背景**：参加頻度上限は本来ジャンル（活動）に紐づく個人的な制約（30章参照）だが、コミュニティによって熱量や付き合い方が異なる実態（例：本拠地のコミュニティでは週1回、たまに顔を出すコミュニティではさらに少なめにしたい）に対応するため、コミュニティ単位の上書きを可能にする。

**入力項目**：上限回数（正の整数、任意）、期間（週/月、任意）。回数と期間は同時に設定・解除する（片方のみの設定は不可）。空欄で解除しF-003側の全体設定へのフォールバックに戻す。

**対応API**：`PUT /communities/{communityId}/members/me/frequency-limit`

### F-109b 自動承認（コミュニティ単位の上書き） **[v1.8新規]**

**概要**：ユーザーはコミュニティごとに、F-003で設定した自動承認設定を個別に上書きできる。未設定の場合は、プロフィール（F-003）側の全体設定にフォールバックする。表示名（F-108）・参加頻度上限（F-109）と同じフォールバック構造。

**利用者**：コミュニティの全メンバー（自分自身の設定のみ変更可能）

**背景**：自動承認は本来個人の判断基準（どのコミュニティのイベントなら都度確認せず参加してよいか）だが、コミュニティによって関わり方の熱量が異なる実態（例：気心の知れた本拠地コミュニティは自動承認、たまにしか参加しないコミュニティは都度確認したい）に対応するため、コミュニティ単位の上書きを可能にする。

**入力項目**：自動承認する/しない（boolean、任意。未設定でF-003側の全体設定へのフォールバックに戻す）

**対応API**：`PUT /communities/{communityId}/members/me/auto-approve`

### F-502 イベント仮確定 **[v1.2修正、v1.8修正：「イベント承認」から改称]**

**概要**：管理者が開催可否を判断する。従来はこの承認と同時にイベントが確定していたが、v1.8よりF-502bによる参加予定者全員の承認を経て本確定する多段階フローに変更した。

**利用者**：OWNER / ADMIN

**仮確定時処理**：
1. **ダブルブッキング事前チェック**（候補メンバー全員について、既に確定済みまたは承認待ちの他イベントと時間が重複していないか確認。重複があれば仮確定をブロックする）[v1.2追加、v1.8で判定対象に「承認待ち」を追加]
2. 候補メンバーごとに実効自動承認設定（F-003/F-109b）を解決
3. イベント状態更新（`AWAITING_MEMBER_APPROVAL`、候補メンバー全員が自動承認ONの場合のみ`CONFIRMED`）[v1.8修正]
4. 参加者作成（自動承認ONのメンバーは確定、それ以外は承認待ち）[v1.8修正]
5. 通知送信

> **[v1.2追加]** ユーザーが複数コミュニティに所属できる仕様上、仮確定の瞬間に実際の時間重複（ダブルブッキング）を機械的に防止する。これは候補生成時のスコアリングとは独立した、確定処理直前の最終ガードとして位置づける（要件定義書v1.2 17章）。
>
> **[v1.8追加]** 仮確定の時点で参加者の時間枠は事実上「予約済み」になるため、ダブルブッキング判定の対象を「確定済み」だけでなく「承認待ち」にも拡張した。

### F-502b 参加承認（全員承認） **[v1.8新規]**

**概要**：参加予定者本人が、仮確定（F-502）後に承認待ちとなっている自分の参加を個別に承認する。候補メンバー全員の承認が完了した時点で、イベント自体が本確定する。

**利用者**：仮確定されたイベントの参加予定者本人

**処理**：
1. 参加者本人の承認状態を確定に更新（既に応答済みの場合はエラー）
2. 全参加者が確定状態か判定
3. 全員確定していれば、イベント状態を本確定（`CONFIRMED`）に更新し確定通知を送信
4. 一部のみの場合は、そのまま承認待ちを継続

> 自動承認（F-003/F-109b）がONのメンバーは、仮確定の時点で確定扱いになるため、本人による個別の承認操作は不要。全参加予定者が自動承認ONの場合、F-502の仮確定時点で本確定まで完了する。

### F-502c 参加辞退 **[v1.8新規]**

**概要**：参加予定者本人が、仮確定後に承認待ちとなっている自分の参加を明示的に辞退する。F-601（個人キャンセル申請）とは異なり承認待ち状態専用で、管理者の承認を待たず即座に辞退が確定する。

**利用者**：仮確定されたイベントの参加予定者本人

**処理**：
1. 参加者本人の状態を辞退済みに更新（既に応答済みの場合はエラー）
2. 管理者へ辞退の通知を送信

> システムはイベントを自動では中止しない（要件定義書v1.7 §17/§19のヒューマン・イン・ザ・ループ原則）。辞退が発生した場合、イベントを継続するか中止（F-605）するかは管理者が判断する。欠員が出た場合の代替メンバーの自動的な差し込みはF-603（欠員補充）と同様Phase2の検討事項とする。

### F-503 イベント状態管理 **[v1.8修正]**

| 状態 | 説明 |
|---|---|
| DRAFT | 作成中 |
| OPEN | 候補受付中 |
| MATCHING | マッチング中 |
| PENDING_APPROVAL | 管理者確認待ち |
| AWAITING_MEMBER_APPROVAL | メンバー承認待ち（管理者の仮確定後、参加予定者全員の承認待ち）[v1.8追加] |
| CONFIRMED | 開催確定 |
| IN_PROGRESS | 開催中 |
| COMPLETED | 終了 |
| CANCELLED | 中止 |

### F-504 イベント詳細表示

**表示内容**：日時、場所、ゲーム種別、参加者、状態、コメント

## 10. キャンセル管理機能 **[v1.1修正]**

> **[v1.1修正]** v1.0では「イベント全体の中止」と「個人の参加辞退」の区別が曖昧だったため、本版で明確に分離する。

### F-601 個人キャンセル申請（MVP）

**概要**：参加者が参加辞退を申請する。イベント全体の中止（F-605）とは別の操作。

**入力**：理由（仕事、体調不良、その他）

### F-602 キャンセル申請承認（MVP）

**利用者**：管理者

> **[v1.1補足]** キャンセル申請は自動受理されず、管理者の承認を経て確定する。これは、既に参加確定している他メンバーへの心理的影響（人数が減ったことによる参加モチベーション低下）を避けるため、管理者が状況を見て調整できる余地を残す設計意図による。

### F-603 欠員補充（Phase2）

**概要**：キャンセル承認発生時、代替参加者を検索する。

**処理**：キャンセル発生 → 不足人数確認 → 候補検索 → 通知 → 参加確定

> **[v1.1補足]** MVP対象外（Phase2）。実装方式（EventBridgeイベント連鎖／Step Functions併用）は実装段階で決定する。

### F-604 繰り上げ候補通知（Phase2）

**通知対象**：補欠ユーザー、管理者

### F-605 イベント中止（MVP） **[v1.1新規]**

**概要**：管理者がイベント自体を開催中止にする。個人のキャンセル申請（F-601）とは別の操作で、全参加者へ中止通知が送られる。

## 13. 設計上の重要ポイント

MeetFlowではマッチング結果を完全自動決定しない。

**理由**：コミュニティには、人間関係、雰囲気、初心者への配慮、メンバー同士の相性など、数値化しにくい要素が存在するため。

そのため、「AI・システムが候補提示」＋「管理者が最終判断」というハイブリッド方式を採用する。

## 22. 開発方針

MeetFlowは初期段階では複雑なAI最適化を行わず、「条件一致によるマッチング」から開始する。

ユーザー利用データを蓄積しながら、信頼度、相性、継続率、満足度を分析し、段階的に高度化する。

---

## 23. 完成イメージ

MeetFlowは単なる予定調整サービスではなく、「リアルなつながりを自然に生み出すコミュニティプラットフォーム」として成長する。

暇な時間を、カラフルに。人と人が会うきっかけを、もっと簡単に。
